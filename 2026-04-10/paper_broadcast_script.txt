各位早上好，今天是二零二六年四月十号。欢迎收听今天的 AI Agent 日报。

如果今天只能记住一句话，那我想先把这个判断放在最前面：Agent 研究的主战场，正在从“会不会做事”，转向“能不能稳定、可控、可恢复地把事做完”。

这不是一个小变化。因为过去一段时间，大家看 Agent，往往先问它能不能跑通任务、能不能刷高 benchmark、能不能展示出很像人的规划和工具调用。但今天这一批值得看的工作，明显把重心放到了另一边：真实环境里的可用性，执行过程的可观察性，以及出错之后怎么诊断、怎么拦截、怎么恢复。

你会发现，评测也在变。以前很多论文更像是看终局分数，任务成没成功，答对了多少题，跑完了几个流程。但现在越来越多工作在强调“轨迹级评测”，也就是不只看最后结果，还要看 Agent 过程中到底看到了什么、点了什么、调了什么工具、在哪一步偏了、偏了之后有没有回正。这个变化非常关键，因为 Agent 的失败，很多时候不是最后一刻突然坏掉，而是前面某个感知或者规划的小误差，一路传导，最后变成整单任务失败。

而且，安全问题也在升级。风险已经不只是“模型说错一句话”这么简单了。现在更大的风险，来自 Agent 读环境、做判断、再调用动作这一整条链路。尤其是 web agent、computer-use agent 这些会直接操作界面和外部系统的 Agent，它们面对的不再只是提示词攻击，还包括界面干扰、注意力劫持、级联注入，甚至是环境本身把它一步步带向错误动作。

所以今天这期日报，我们重点看三篇论文。它们表面上分别在讲代码 Agent、系统可靠性、还有网页 Agent 评测，但底层其实在回答同一个问题：Agent 到了真实世界，怎么从“结果黑盒”变成“过程可分析”的系统。

先插一个今天很值得听懂的概念，叫轨迹级评测，也就是 Trajectory Evaluation。

你可以把它理解成一种“全过程体检”。不是只看 Agent 最后有没有完成任务，而是沿着它的执行轨迹逐步检查：它看到的信息对不对，理解得对不对，计划是不是合理，点击和输入是不是正确，工具调用有没有副作用，出了错之后有没有识别、有没有恢复。为什么这件事这么重要？因为如果你只看成败，很多时候你根本不知道问题出在哪。是模型没看懂页面？是定位代码的能力差？是工具返回了模糊结果？还是系统没有设计恢复机制？轨迹级评测的作用，就是把“这系统行不行”进一步拆成“它到底坏在哪、修完有没有真的变好”。今天三篇论文，其实都在往这个方向推进。

先看第一篇，ORACLE-SWE。

这篇论文我觉得特别值得做代码 Agent 的团队认真读，因为它没有再做一个新的 SWE Agent 来比谁分高，而是干了一件更基础也更有指导意义的事：它去量化，如果给代码 Agent 某种“完美线索”，到底能带来多大提升。

这个问题非常重要。因为今天很多代码 Agent 的流程都差不多：读 issue，搜仓库，定位文件，改代码，跑测试，反复修。但现实里大家常常同时优化很多模块，最后成功率涨了，你也不一定知道到底是哪块带来的；成功率不涨，也不清楚真正的瓶颈卡在哪。ORACLE-SWE 的办法是把常见的关键信息拆开来，一类一类做 oracle 注入实验，也就是假设某类信息是完美的，然后看它能把 Agent 成功率抬高多少。

论文拆了五类信号：复现测试、回归测试、编辑位置、执行上下文，还有 API 使用信息。通俗讲，就是它把“修 bug 需要的线索”拆成了几种不同类型，再分别喂给 Agent，看哪一种最值钱。

结果非常清楚，最重要的不是别的，而是复现测试。也就是说，对代码 Agent 来说，最有价值的线索，不是先告诉它改哪个文件，也不是先给它一堆 API 文档，而是先把问题稳定、准确地复现出来。这个结论很有启发。因为它说明很多时候 Agent 真正缺的不是“写 patch 的手”，而是“把问题说清楚的任务书”。

你可以想象一下，一个 issue 只写着“某些输入下结果不对”。这对 Agent 来说其实很模糊，它得先自己猜用什么输入，怎么跑，哪里会错，正确行为又是什么。但如果你直接给它一个高质量的 reproduction test，也就是一段能稳定失败的测试，那么错误行为、触发条件、预期输出，就都被明确了。Agent 接下来做的事情，就从“在雾里摸索”变成“沿着失败样例找原因”。

论文还发现，执行上下文有帮助，但这个帮助是有条件的。最有价值的上下文，是运行时天然产生的错误堆栈，也就是那种能直接告诉你哪里抛错、调用链怎么走到这里来的证据。相比之下，如果拿不到原生错误堆栈，只能靠人工插桩去补一些调用路径信息，帮助就会小很多。原因也很直观：原生报错通常会告诉你“为什么错”，而补出来的调用路径很多时候只是在告诉你“可能在哪一带出事”。

另一个很重要的发现，是多种线索组合起来的效果很强。测试告诉你问题是什么，编辑位置告诉你去哪里改，API 使用信息告诉你该怎么接接口，上下文再补足调用链，这些东西叠起来，成功率就不是简单相加，而是互相放大。论文甚至在 oracle 条件下观察到非常高的上限，这说明对 SWE 任务来说，关键不只是模型本身聪不聪明，而是系统有没有把中间线索组织好。

我觉得最有工程价值的一点，是它还做了一个更现实的验证：先让强模型提取关键信息，再让弱模型去修 bug。结果和 oracle 排序基本一致，尤其再次证明复现测试极其关键。这个结论对产品很实用，因为它暗示一种更划算的架构：不一定非要让一个最贵的大模型从头做到尾。更现实的方式，可能是让强模型负责“侦察”和“提线索”，比如生成复现测试、定位关键位置；然后再交给更便宜的执行模型去改代码、跑验证。

所以如果你今天在做代码 Agent，我觉得这篇论文给出的优先级很明确。第一优先不是继续堆一个更复杂的 agent loop，而是提升问题复现能力，尤其是生成高质量复现测试、最小失败样例，以及把失败日志结构化。第二，是把定位、上下文、API 检索这些信息做成中间产物，而不是让主 Agent 每一轮都从零开始搜。

接下来第二篇，LogAct。

如果说 ORACLE-SWE 在问“代码 Agent 缺什么信息”，那 LogAct 问的是“Agent 运行时怎么才能更可靠”。这篇论文我觉得特别像一篇面向生产环境的基础设施论文。它的核心改动听起来很简单，但影响很大：Agent 不再是边想边做、想完就执行，而是先把“准备做什么”写进共享日志，再由别的组件投票、审批，最后才真正执行。

这个设计，等于把传统的黑盒循环拆开了。

以前的 ReAct 或 CodeAct 式 Agent，很像一个人边想边按按钮。你很难在它出手之前拦住它，也很难在出错之后还原它当时到底怎么想的。LogAct 的思路是，给 Agent 加一条共享日志总线。模型先写 intention，也就是“我下一步打算执行什么动作”；然后投票器和决策器先过一遍，看这动作是不是安全、是不是符合策略；只有在日志里写下 commit，也就是批准之后，执行器才真正去操作环境。

这个变化的价值，主要在三个词上：可审计、可拦截、可恢复。

先说可拦截。比如用户让 Agent 做一个线上运维任务，模型下一步想删数据库。在传统循环里，这条命令可能直接就出去了。但在 LogAct 里，它得先作为 intention 暴露出来，安全组件先看一眼。如果规则判断这是危险动作，就直接 abort，执行器因为没收到 commit，也就不会真正下手。也就是说，风险有机会在动作发生前被拦住，而不是事后补救。

再说可审计。论文里这条共享日志叫 AgentBus，本质上是一条有类型、可持久化、追加式的系统总线。不同组件围着它协作，但每个组件能读什么、写什么都受限制。Driver 负责调用模型、生成意图；Voter 负责安全审核；Decider 负责拍板；Executor 负责执行。整个过程都在同一条日志上留下记录，所以任务结束后你可以回放整条链路，知道它何时产生了什么意图，谁投了反对票，哪一步被放行，哪一步失败。

更关键的是可恢复。现实里的 Agent 往往不是在一个干净、可回滚、完全可重放的世界里运行。它可能搬一半文件崩了，改一半配置断了，调一个外部 API 返回了未知状态。传统恢复手段往往要求动作足够规则、足够确定，但 Agent 执行的东西常常并不满足这些条件。LogAct 的思路是，既然没法靠纯规则恢复，那就先把历史轨迹完整记下来，再让 Agent 读自己的日志、查看当前环境，去推断“我断在了哪一步，哪些已经做过，哪些还没做”。这更像运维排障，而不是简单 checkpoint 回滚。

论文还有一组很有意思的实验，放在 AgentDojo 这种安全评测环境里。它用规则型投票器加上 LLM 型投票器做双重审核，结果是既能把有害动作大幅拦住，又能降低纯规则系统的误杀。这个结果说明一个很现实的工程方向：Agent 安全不是只能靠“换更聪明的大模型”，还可以靠一个可插拔、可叠加的控制面。规则负责兜底，LLM 审核负责理解上下文，两者组合起来更像一条能逐步迭代的安全工程路线。

当然，这篇论文也不是在说它已经解决了所有问题。LLM 做内省和恢复，本身也可能判断错；并发、外部环境突变、投票延迟和 token 成本，这些问题依然存在。但它的价值在于，它把 Agent 从“单体循环”推进到了“日志驱动的状态机”。对于任何要接工具、写代码、改外部状态的 Agent 来说，这都是一个很值得关注的运行时方向。

第三篇，ClawBench。

这篇论文我会把它看成今天“真实世界差距”的最直接提醒。因为它不是在一个简化沙盒里测网页 Agent，而是直接问：Agent 到底能不能在真实网站上完成日常事务。

很多现有 Web Agent benchmark，为了可控和可复现，都会把环境做得比较干净。页面相对静态，DOM 结构固定，没有登录态，没有太多动态内容，也不会真的产生线上副作用。这样的好处是容易评测，但坏处也很明显：真实网页里最麻烦、最影响可用性的东西，很多都被拿掉了。比如 cookie 弹窗、动态渲染、多步骤流程、长表单、上传文件、复杂筛选，还有提交前那一连串状态确认。

ClawBench 就是要把这些东西重新加回来。它构建了一百五十三个真实在线任务，覆盖一百四十四个真实平台，任务类型也非常贴近日常生活和工作场景，包括购买、预约、申请、订票、填表、投递之类。这一点很重要，因为它考的不是“会不会看网页找答案”，而是“能不能真的把事办成”。

论文最巧妙的设计，是它没有为了安全把真实网站整个封掉，而是只拦最后那个真正会产生不可逆后果的提交请求。也就是说，Agent 前面该点就点，该填就填，该翻页就翻页，整个体验和真人差不多；只有到了最后一步，比如下单、提交申请、发送正式请求的那个 HTTP 调用，它才会被系统拦下来。这样一来，评测既保留了真实网站的复杂性，又不会真的在网上乱下单、乱提交。

更重要的是，ClawBench 做了很完整的轨迹记录。它不仅录视频、截屏，还记录 HTTP 流量、Agent 消息、工具调用、底层浏览器动作。然后把 Agent 的整条轨迹和人工完成同一任务时留下的人类参考轨迹进行对照。也就是说，它不是只看最后页面像不像，而是尽量判断：关键字段填对了吗，动作顺序是否合理，流程是否等价，停在某个验证环节之前算不算完成。

这就是为什么今天开头我会强调轨迹级评测。ClawBench 不是简单告诉你“成功率只有多少”，它更像是在告诉你，Web Agent 到底是在哪里掉链子的。

而结果，确实很能泼冷水。论文里最强模型在这个真实网站任务集上的成功率，也只是三成出头。更值得注意的是，一些在传统 benchmark 上表现不错的模型，到了 ClawBench 上成绩明显下滑。这说明我们之前可能高估了 Web Agent 的真实可用性。沙盒里的优等生，不等于现实世界里的办事高手。

这个落差从哪里来？其实很容易理解。真实网页里，Agent 可能先被 cookie 弹窗打断，然后遇到动态加载，接着进入一个多页长表单，中途还要上传文件、理解字段约束、处理反机器人机制。你在某个 benchmark 里看起来只是“多走了几步”，但在真实环境里，每一步都可能引入状态偏差。最后只要有一个字段填错、一个下拉选错、一个日期格式不对，整单任务就挂了。

所以这篇论文对产品的启发非常直接。如果你想做真正有商业价值的 Web Agent，目标不应该只是“能浏览网页”，而应该是“能完成事务”。这意味着系统必须具备跨页面长流程处理、表单精确填写、弹窗与异常恢复、提交前确认，以及网络层的安全拦截能力。否则看起来像在做自动化，实际上离真实可用还很远。

把今天这三篇放在一起看，你会发现它们共同指向一个很清晰的方向。

ORACLE-SWE 告诉我们，Agent 的瓶颈往往不是最后那一步推理，而是中间信息有没有被正确抽取和组织。LogAct 告诉我们，Agent 的可靠性不能只靠模型聪明，而要靠运行时结构，把意图、审核、决策、执行拆开。ClawBench 告诉我们，很多看起来不错的 Agent，一旦放进真实环境，问题会在过程里被放大，而不是在最终分数里自动暴露。

所以今天最值得带走的判断，我想用三句话来收束。

第一，Agent 研究正在离开“单次任务跑通”的阶段，进入“真实环境里能不能被测、被管、被恢复”的阶段。

第二，无论是代码、网页，还是更广义的工具使用场景，大家都在补同一块短板：把执行过程结构化。只有过程被记录、被分析、被切分，我们才有资格讨论可靠性、安全性和长期能力。

第三，如果说过去一年 Agent 的核心问题是“它会不会做事”，那么现在更关键的问题已经变成：“它为什么这样做，出了错怎么发现，以及我们能不能把它稳定地拉回正确轨道。”

以上就是今天的 AI Agent 日报。我们明天见。
