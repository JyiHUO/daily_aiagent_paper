各位早上好，今天是二零二六年四月九日，欢迎收听今天的 AI Agent 日报。

如果今天只记住一个趋势，我觉得应该记住这句话：Agent 研究的重心，正在从“展示能力”转向“建设可靠性基础设施”。换句话说，社区现在不再满足于让 Agent 看起来能做事，而是越来越在意，系统能不能被可信地判断做对了，做错了，以及到底错在哪里。

这个变化，今天特别明显。高关注论文几乎都围绕 verifier，也就是验证器，benchmark，也就是评测基准，trajectory guardrail，也就是轨迹护栏，还有系统性安全模型展开。这背后其实是一个很重要的分水岭：大家开始明确区分“模型会做”，和“系统能被可靠监督地做成”是两回事。前者更像 demo，后者才更接近产品。

另一个升温很明显的方向，是 computer use，也就是电脑操作和 GUI Agent。过去很多工作更像 screen-to-action，看到屏幕然后点按钮。但今天的信号已经变了，研究焦点在往 UI-in-the-loop、移动端真实诊断基准、以及任务完成验证迁移。也就是说，界面型 Agent 正在从“能操作”走向“能在真实产品约束下稳定完成任务”。

同时，安全方向也在变复杂。风险已经不只是单轮 prompt 的提示注入，而是跨工具、跨模块、跨执行链的组合脆弱性。今天能看到，提示注入、防御 wrapper 失效、技能层后门、多步工具轨迹护栏，这些问题被放到同一个视野里讨论。这说明 Agent 的攻击面，已经是系统级的了。

还有一个值得注意的研究口味变化，是规划和训练不再只是追分，而是开始盯着能力上限和失稳机制。比如 latent planning 的深度天花板，比如今天会讲到的 agentic RL 里的 reasoning collapse，也就是推理塌缩。大家越来越意识到，长程 Agent 的瓶颈，很多时候不是再加一点数据、再堆一点技巧就能解决，而是系统性的限制。

在进入论文之前，先补一个今天特别值得懂的小知识点，叫 trajectory evaluation，轨迹级评估。

简单说，它不是只看 Agent 最后答案对不对，而是看整条执行过程。它会把任务拆成状态、动作、工具调用、环境反馈和中间决策，去检查每一步是不是合理，是不是可恢复，是不是可解释。为什么今天要特别讲这个？因为你会发现，今天最重要的论文，无论是 verifier、computer-use 任务完成判定，还是推理塌缩诊断，都不能只靠最后一个结果分来理解。很多失败，看最后答案你以为只是“没做对”，但沿着轨迹看，你才能分辨它到底是看错了界面，计划错了步骤，调用错了工具，还是被外部内容带偏了。

所以现在 Agent 研究正在从“答案评测”转向“过程评测”。这不是学术细节，而是部署门槛。因为真实世界里的 Agent，不是一次性答题器，而是一条长执行链。

接下来讲今天三篇最值得听的论文。

第一篇，是今天我认为最值得技术团队重点关注的一篇，叫 RAGEN-2，主题是 agentic RL 里的 reasoning collapse，推理塌缩。

这篇论文讨论了一个很隐蔽、但可能已经在很多训练系统里发生的问题。我们以前看多轮 Agent 强化学习训练，通常会看 reward，也就是结果分数；或者看 entropy，也就是输出多样性，来判断模型有没有塌掉。但这篇论文说，这两种看法都不够。因为有一种更危险的情况是，模型表面上还在“认真推理”，话很多，形式也丰富，熵看起来不低，但它实际上已经越来越不看输入了，开始套固定模板。

这是什么意思？你可以把它想象成，模型学会了“像在推理”，但不再“根据任务去推理”。不同问题来了，它先给你一段相似的分析框架，再走一套通用动作流程。措辞上似乎有变化，所以传统指标可能还觉得它挺活跃，但真正和输入绑定的那部分推理，已经在变弱。

论文把这种现象叫 template collapse，模板塌缩。它的核心不是输出变少了，而是输入依赖性变低了。这个判断很关键，因为它直接改变了我们监控 Agent 训练的方式。

作者提出，比起单纯看熵，更应该看 mutual information，也就是输入和推理轨迹之间的互信息。通俗说，就是问这样一个问题：看一段推理过程，你还能不能猜出它原来是为哪个输入写的？如果能猜出来，说明推理真的和输入绑定；如果放到很多任务上都差不多说得通，那就是模板化了。

这个思路非常漂亮，也很实用。因为它抓住了一个本质：真正的推理能力，不是话多，不是格式完整，而是遇到不同环境状态时，能不能给出不同且贴题的思考过程。论文里给出的 MI 代理指标，和最终任务表现的相关性明显比 entropy 更强。这意味着，以后很多训练团队如果还只盯着成功率、长度、格式合规率和熵，可能会错过非常早期的退化信号。

那为什么会发生这种塌缩？论文进一步给了一个很有启发的解释：低信噪比。也就是训练里真正有用的任务梯度太弱，而那些和输入内容无关的正则化力量，比如 KL 约束、熵正则，还在持续作用。久而久之，模型更容易学会统一、安全、流畅的“推理模板”，而不是针对不同状态做差异化思考。

这个解释很像工程现场的感受。很多时候系统不是突然坏掉的，而是慢慢失去针对性。表面看，格式还在，链路还在，甚至输出还挺长，但模型已经不再认真看环境了。

这篇论文最有价值的地方，还在于它不只诊断问题，也提出了一个很轻量的干预方案，叫 SNR-Aware Filtering。思路是这样的：每轮 rollout 之后，先根据每个 prompt 的 reward variance，也就是奖励方差，估计这个样本组的信号强度。只保留那些真正有信息量的高信号样本来更新参数，低信号的样本就先不过度学习。相当于训练时不是所有样本一视同仁，而是优先学那些“真的能教会模型东西”的样本。

这个方法的好处是成本不高，不需要额外模型，也不太增加 rollout 负担。对训练系统来说，这很现实。因为很多团队不是缺概念，而是缺能挂进现有管线里的东西。

如果从产品和系统角度看，这篇论文的启发很直接。第一，不要把“会写推理过程”等同于“会根据任务推理”。第二，线上监控应该增加输入依赖性的指标，比如 reasoning-to-input 的可检索性，或者 MI proxy。第三，训练平台可以把 reward variance 这种轻量信号纳入稳定性面板，作为早期预警的一部分。

一句话总结这篇论文：Agentic RL 真正危险的塌缩，可能不是不说话，而是说得越来越像，但越来越不看你在说什么。

第二篇重点论文，是 computer use 方向今天最重要的一篇，题目叫 The Art of Building Verifiers for Computer Use Agents。

如果说前几年 computer use 的问题是“Agent 会不会点按钮”，那现在最痛的问题其实变成了：它到底有没有真的完成任务，以及我们能不能可靠地判断这件事。

这篇论文切中的就是这个核心。因为电脑操作类 Agent 有几个天然难点。第一，轨迹很长。第二，它靠视觉界面推进。第三，成功标准常常有歧义。比如加入购物车算完成吗？必须下单才算吗？如果卡在登录墙前面，是 Agent 失败，还是环境阻断？这些问题都让“任务有没有完成”比普通文本任务复杂得多。

过去很多做法要么只看最后结果，要么把大量截图一股脑塞进上下文，让模型自己总结。但这两种方式都很容易出错。只看最后，会漏掉中间关键证据；全塞进去，又容易上下文过载，还可能被 Agent 自己编造的最终描述带偏。

这篇论文提出一个核心观点：好的 verifier，第一步不是找最强的模型，而是先把评分标准，也就是 rubric，写对。

这个点我很认同。因为 verifier 本质上是阅卷系统。题目都没出好，模型再强也只是高水平误判。论文强调，rubric 必须具体、互不重叠、紧扣任务目标，而且要先从任务本身生成，不要看完轨迹再倒推标准。这样才能避免因为 Agent 走了不同但合理的路径，就被随意扣分。

它还特别处理了条件式任务。比如“买有机蓝莓，如果没有就买非有机蓝莓”。这种任务里，不是所有标准同时生效，而是看执行过程中现实条件触发哪一条。这个设计很重要，因为真实产品环境里，任务很多都不是单一死规则。

第二个关键点，是把过程和结果分开判。论文把每条轨迹拆成 process reward 和 outcome reward。前者看过程做得怎么样，后者看最终用户目标是不是实现了。为什么这很关键？因为在真实环境里，Agent 可能路径完全合理，但被库存、登录、验证码这种外部条件卡住；也可能它走了很奇怪的路径，但最后事情办成了。如果你只给一个成败标签，很多信息就丢了。

第三个很实用的设计，是区分可控和不可控失败。网站没货、需要账号、服务不存在，这些很多不是 Agent 能控制的；而选错对象、没用可用筛选器、或者明明失败了还瞎报成功，这些才是 Agent 的责任。这个切分对产品特别重要，因为它能告诉你，接下来该补的是模型，还是产品流程，还是环境兼容性。

而且论文很明确地提醒一点：别只看最后几张截图。长轨迹验证里，最容易被忽略的就是中间关键状态。所以他们的方法是，先建立“每条标准对应哪些截图最相关”的关系，再为每个 criterion 选最相关的 top-k 证据图去判。这种做法本质上就是轨迹级评估的一个非常具体的落地：不是让模型泛泛回忆，而是按检查点去看证据。

还有一个特别像真实线上风控需求的设计，是专门检查副作用。因为很多时候 Agent 看似完成主任务，但顺手做了用户没授权的事。比如多加了延保，多勾了服务，多放了一件商品进购物车。对用户来说，这不叫“差不多完成”，而是失败，甚至是风险事件。论文把这个单独拉出来做 side-effect 检测，我觉得非常对。

这篇论文的重要性在于，它把 verifier 从一个“附属打分器”，提升成了 computer use 的基础设施。它不只服务评测，也服务训练闭环、自动回放、奖励建模和线上风控。尤其论文提到，在 outcome 误报成功这件事上，已经能压到接近零。这个价值非常大。因为对操作类 Agent 来说，最危险的不是不会做，而是明明没做成，却被系统判成做成了。

所以这篇论文背后的判断是：computer use 真正要走出 demo 阶段，不是先把动作做得更炫，而是先把“是否真的完成”这件事判准。

第三篇重点论文，转到安全方向，题目叫 SkillTrojan，研究的是 skill-based agent systems 上的后门攻击。

这篇论文提醒了一个很多团队可能还没真正重视的风险层：不是模型，不是提示词，而是技能包本身。

现在很多 Agent 系统越来越模块化，喜欢把能力封装成 skill，技能，或者插件、工具包。这样做的好处是复用性高、工程结构清晰、好编排。但代价是，信任被下沉到技能这一层。过去大家防安全，主要盯模型参数、训练数据、prompt、记忆、工具描述。可这篇论文说，一个完全没被污染的底层模型，只要装进了看起来正常的第三方技能，也可能在正常工作流里触发隐藏恶意行为。

它的攻击思路很像供应链后门。攻击者伪装发布一个“有用且正常”的技能，技能的说明文档和执行脚本都看起来没问题。开发者把它装进系统后，平时跑起来一切正常，clean accuracy 也不差。但一旦遇到特定触发条件，这个技能会在标准编排流程里偷偷执行预设 payload，也就是恶意载荷。

更厉害的是，这个 payload 不是明着塞进去，而是被拆成多个片段，分散埋进多次看似正常的技能调用之中。单看某一次调用，你只觉得输出里多了一点不起眼的元数据；但等整轮任务跑完，系统里的专门验证动作就能把这些碎片重组、解密、执行。也就是说，恶意行为不是某一步突然爆出来，而是沿着整条执行链悄悄完成。

这就解释了为什么传统行为测试不一定能发现问题。因为用户看到的最终答案可能还是对的，Agent 表面行为也很正常。危险在于“表面正确，背后越权”。这对数据库访问、文件操作、代码执行、外部 API 调用场景特别致命。

论文还有一个更值得警惕的点：这不是孤例攻击，而是可以规模化自动生成。作者给出了从公开技能模板批量合成后门技能的流程，还构建了一个大规模后门技能集合。换句话说，问题已经不是“某个插件可能有问题”，而是整个技能生态都可能变成供应链风险面。

如果从系统建设角度看，这篇论文给的启发非常明确。第一，技能不能被默认视为可信模块。第二，技能市场、第三方插件接入、企业内部技能仓库，都要做来源验证、版本审计、权限声明和行为留痕。第三，单纯做输入清洗或者盯模型输出不够，必须约束技能执行环境本身，比如沙箱、最小权限、调用链审计、异常中间字段检测。

所以这篇论文其实在扩展我们对 Agent 安全的理解：风险不只在“说什么”，更在“做什么”，以及“通过哪个可复用模块去做”。

最后收束一下今天的判断。

今天的论文分布非常清楚地说明，Agent 研究正在进入一个更工程化、也更严肃的阶段。前沿不再只是让模型多会一点，而是让系统知道自己什么时候做对，为什么做错，哪里会失控。

从 computer-use verifier，到多步轨迹评估，再到 agentic RL 里的推理塌缩，大家其实都在回答同一个问题：长链条 Agent，怎样才能拥有可诊断、可监督、可部署的运行机制。

如果说上一阶段比的是“Agent 能不能做事”，那么这一阶段开始比的，就是“Agent 能不能在复杂环境里稳定、可信地把事做成”。

而今天最值得带走的一句话，我想再重复一次：Agent 的竞争焦点，已经从能力演示，转向可靠性基础设施。
