各位早上好，今天是二零二六年四月八日。欢迎收听今天的 AI Agent 日报。

如果今天只记住一个趋势，我会建议你记住这句话：Agent 研究的重心，正在从“谁更聪明”转向“谁更可靠”。

什么意思？不是说大家突然不关心模型能力了，而是社区越来越清楚地意识到，一个 Agent 真正能不能上线，不取决于它偶尔做出多惊艳的演示，而取决于它能不能在真实环境里稳定执行、可重复比较、出了错能恢复、遇到风险能收住。换句话说，现在的关注点，正在从能力展示，转向系统验证。

今天这个信号特别明显。高分论文里，最集中的不是“又一个更强的通用 Agent”，而是评测基础设施、生产力工作空间、GUI 闭环执行，以及安全红队。这里面其实透露出同一个判断：大家默认 Agent 的核心难题，已经不只是会不会想，而是能不能在真实软件、真实工具链、真实约束下，把事情做成，而且别做坏。

还有第二个很重要的变化。Agent 安全，正在从过去盯着 prompt 输入，扩展到整条决策链路。也就是说，风险不再被理解成“用户说了一句危险的话，模型回了一句危险的话”，而是记忆、检索、规划、工具调用、执行，这一整条链条都可能被操控。这个变化非常关键，因为它意味着防护思路也要跟着升级。

第三个信号，是 computer-use 和 GUI Agent 明显进入了闭环执行阶段。过去很多系统主要回答“下一步该点哪里”，现在更重要的问题变成了：“刚才点的那一下到底生效没有？如果没生效，能不能意识到、纠正、继续推进？”这听起来像个细节，但它实际上很可能就是实验室 demo 和产品落地之间的分水岭。

在进入论文之前，先插一个今天最值得懂的概念：脚手架，也就是 harness。

你可以把 harness 理解成套在基础模型外面的那层 Agent 运行框架。模型本身会说话、会推理，但 harness 决定它什么时候看环境、怎么拆任务、怎么调用工具、怎么读写记忆、怎么检查结果、什么时候停止、失败了怎么回退，以及哪些动作是被安全约束住的。它有点像 Agent 的操作系统。

为什么今天特别值得讲这个？因为今天三篇重点论文，其实都在反复说明同一件事：Agent 最终表现，往往不是模型参数单独决定的，而是这层外部脚手架怎样组织观察、决策和执行决定的。尤其第一篇论文，几乎是直接把这个结论摆在台面上：很多时候，脚手架设计对结果的影响，甚至可能大过模型差异本身。

好，下面进入今天的三篇重点论文。

先说第一篇，也是今天最值得产品团队关注的一篇，ClawsBench。它的标题翻成大白话，就是：给生产力 Agent 造了一个高仿真的办公沙盒，用来同时评估它“会不会做事”和“会不会惹事”。

为什么这篇特别重要？因为过去很多 Agent benchmark，环境都太干净了。任务是抽象的，工具是简化的，状态是临时的。模型只要大致走对流程，看起来就像“完成了任务”。但现实里的生产力 Agent，不是做一道题，而是要去碰邮箱、日历、文档、网盘、团队聊天，还要跨服务协作，还要处理权限、共享、修改、外发这些高风险动作。你不把这些放进环境里，很多评测其实都离上线很远。

ClawsBench 做的，就是搭了一个假的、但很像真的办公套件。里面有邮件、有聊天、有日历、有文档、有云盘，而且这些服务不是摆设，是有状态的、能回滚的、可以快照恢复的。Agent 不是在那儿“描述自己会怎么做”，而是真的去调接口、改状态、发消息、改文档。最后评测也不是看它生成的文字像不像对，而是直接检查环境状态有没有按要求被正确修改。

这个思路特别重要。因为它把评测从“说得像”升级成了“做得对”。

比如一个任务是：整理一段邮件线程，把结论同步到指定频道。过去你可能看模型生成了一段总结，就觉得差不多了。但在 ClawsBench 里，系统会真的去看：邮件状态有没有变，Slack 指定频道里有没有消息，内容字段对不对，发没发错地方。如果发错频道、改错对象、或者多做了不该做的动作，都会被抓出来。

更关键的是，这篇论文把能力和安全拆开评了。这个设计我觉得非常值得行业借鉴。因为很多系统现在还喜欢用一个总分去概括一切，但现实不是这样。一个 Agent 什么都不做，很安全，但没用。另一个 Agent 很能干，但顺手把敏感文件共享出去了，那你也不能说它表现好。

所以作者把“能不能完成任务”和“有没有执行危险动作”分开了。这样你就能看清楚，一个系统到底是谨慎地完成任务，还是靠保守拒绝来显得安全，又或者是高完成率但高风险。

这篇论文最扎心的地方，是它明确显示：脚手架设计对结果的影响非常大。作者把脚手架拆成两部分，一部分像分服务的技能手册，告诉 Agent 这些工具怎么用、接口长什么样、哪些参数必须带；另一部分像跨服务的元规则，约束它怎么处理权限、怎么防注入、怎么避免越界操作。结果是，加上完整脚手架之后，多数模型的任务成功率都有很明显提升。

这说明什么？说明很多时候，Agent 不是纯粹“不会”，而是“不会被正确组织起来”。

同时，这篇论文也给了一个很现实的提醒：强模型不等于更安全。有的模型任务成功率更高，但危险动作率也更高；有的模型看起来更稳，但其实只是更爱拒绝。也就是说，能力、帮助性和安全性，不能用一个总分简单合并。

如果你在做办公 Agent、企业助手、邮件助理、自动排期、文档协同这一类产品，这篇论文最大的启发就是：上线前必须有接近真实工作空间的可回放测试环境，而且评估指标不能只有完成率，至少还要有危险动作率和安全完成率。否则你看到的“效果不错”，很可能只是 demo 层面的错觉。

第二篇论文，讨论的是 GUI Agent，也是今天落地感最强的一篇，题目叫“不要盲目行动”。它要解决的问题其实特别真实：很多 GUI 自动化 Agent 在执行时，默认自己上一步已经成功了，于是页面没变，它还继续点、继续输、继续往下走，最后进入死循环。

这几乎是所有做 computer-use、浏览器自动化、手机助手、企业流程自动化的人都会遇到的问题。原因也不复杂，真实界面并不像 benchmark 那样乖。网络会慢，页面会延迟，弹窗会突然出现，坐标会偏，元素会遮挡。结果就是，模型虽然给出了一个“理论上正确”的动作，但实际根本没生效。

这篇论文最核心的贡献，不是再搞一个更强的规划器，而是给 GUI Agent 补上了闭环：每做一步，都要先预期这一步成功后界面应该怎么变化；下一帧回来之后，先验证这个变化有没有发生；如果没发生，就承认失败，然后纠正。

这个思路其实非常像人。我们点一个按钮，不会默认它一定生效，而是会扫一眼页面，看是不是跳到了下一页。如果没跳，就会重新点、换位置、等加载、或者判断是不是被弹窗挡住了。论文做的，就是把这种动作后核验机制显式加到 Agent 里。

作者提出的框架里，每一步不只是输出“要做什么动作”，还会输出“我预期接下来界面会变成什么样”。下一轮进来，系统先把真实画面和上一步预期做比对。如果没对上，就把它判成失败，而不是继续自我催眠。

为什么这个机制重要？因为 GUI Agent 最大的问题之一，不是做错一步，而是做错了还不知道。真正致命的是后面的连锁错误。它以为自己已经进了某个页面，实际上还停在原地，于是后续所有点击都变成乱点。最后用户看到的就是一个会一直忙、但始终干不成事的系统。

这篇论文还有个很巧妙的地方：它不是完全依赖昂贵的在线环境训练，而是大量利用离线数据，甚至人工构造失败样本来教模型“如何失败后恢复”。这一点很务实。因为现实中，很多训练数据都是成功轨迹，模型只看过“老师每一步都做对”的世界，自然学不会自救。作者专门给它喂“动作发生了但页面没变”的案例，让它学会说：不对，刚才可能没点中，我要重新判断。

而在评测上，这篇论文也很值得学。它不只看最后任务成功率，还专门看两个指标：会不会陷入循环，以及失败后能不能恢复。这个方向我觉得非常对。因为一个 Agent 就算最终偶尔能做成事，如果中途经常卡住、重复点击、不断误操作，产品体验也是灾难。相反，一个系统哪怕终局成功率还没夸张领先，但它更少进入死循环、更会自我修复，用户感受往往会好得多。

所以对做 GUI Agent 的团队来说，这篇论文最重要的结论不是“多了一个新模型”，而是“闭环验证应该成为默认配置”。以后再做 computer-use，如果还停留在“看图然后猜下一步动作”，其实已经有点落后了。真正应该进入标准能力表的，是动作效果验证、失败检测、恢复策略，还有围绕这些设计的监控指标。

第三篇论文讲安全，而且我认为是今天最值得安全团队重视的一篇。标题翻成中文，大概是：别再只盯着 prompt 了，Agent 真正危险的地方在推理链路可能被劫持。

这篇论文针对的是一个很现实的问题。过去很多 red teaming 或 jailbreak 方法，主要思路都是改用户提示词。也就是在输入里塞一些花样，诱导模型越界。但 Agent 和普通对话模型不一样，它有记忆、有检索、有候选重排、有规划、有工具调用。攻击面已经不只是输入框，而是整个内部决策流程。

这篇论文提出的方法叫 JailAgent。它最特别的一点是，不改用户原始输入。用户的问题可以完全正常，表面上看不出异常。但攻击者会去影响 Agent 背后的记忆和决策链路，比如让某些触发词更容易勾出被污染的记忆，让重排器偏爱有毒候选，或者让推理在关键节点被悄悄带偏。

这个思路的危险性在于，它更接近真实生产风险。因为很多防护机制现在都还是围绕 prompt 做的：输入过滤、敏感词检测、提示词审计。但如果攻击不发生在输入端，而是发生在记忆写入、检索命中、候选排序这些中间环节，外部看起来就会非常干净。用户说的是正常话，Agent也像在正常工作，但内部已经被操控了。

论文的方法可以概括成三步。第一步，找出哪些词或语义片段最容易影响模型后续判断。第二步，围绕当前上下文训练一个轻量的重排机制，让系统更容易选中攻击者想让它选的候选。第三步，再通过一组约束优化，让这些触发模式在不同场景下更稳定，不至于换个数据集、换个模型就失效。

作者做了跨多个 Agent 类型、多个模型核心和多个数据集的实验，结论是这种链路级攻击相比一些传统基线更强，而且还能尽量保持原任务表面上不崩。注意，这一点非常关键。真正危险的攻击，不是把系统一下打挂，而是让系统继续“看起来正常”，但已经在一些关键判断上偏离了。

这篇论文对行业最大的提醒就是：Agent 安全不能再只做输入侧防守了。只过滤 prompt，不检查记忆来源，不审计重排过程，不回溯关键工具调用，那很可能只防住了最显眼、最原始的一层风险。真正要防的是整条链路：记忆是谁写进去的，检索为什么命中了这一条，重排为什么给它更高权重，规划为什么走向了这一步，执行结果有没有越界。

把今天三篇论文放在一起看，会发现它们其实都在讲同一件事，只是从三个不同角度切入。

ClawsBench 说的是，没有接近真实环境的评测，你不知道 Agent 到底会不会做事，也不知道它会不会做坏事。GUI 那篇说的是，没有动作后的核验和自纠闭环，你的 Agent 就会活在“我以为我成功了”的幻觉里。安全那篇说的是，如果你还只盯着输入 prompt，就会忽略真正可能被劫持的内部决策链。

而把这三件事再往上一层抽象，它们共同指向的，就是今天开头说的那个判断：Agent 研究，正在从“生成更聪明的下一句话”，转向“构建更可靠的执行系统”。

这里面，脚手架就是那个最容易被低估、但越来越关键的系统层变量。环境怎么组织，动作怎么抽象，工具怎么封装，结果怎么验证，失败怎么恢复，风险怎么约束，这些东西合在一起，往往比换一个更大的底座模型，更能决定一个 Agent 产品最后能不能成立。

所以，如果你是做 Agent 产品、Agent 平台或者企业自动化的，今天最值得带走的实操判断大概有三个。

第一，不要再只看模型分数和任务成功率，要同时看环境真实性、状态级验证、危险动作率、恢复能力和循环率。

第二，把闭环执行做成系统默认项。不要让 Agent 只负责“生成下一步”，要让它对上一步负责，验证动作效果，必要时自我修正。

第三，把安全边界从输入框扩展到整条决策链。记忆、检索、重排、规划、工具执行，都应该纳入红队和审计。

最后用一句话收束今天的日报：现在真正值得投入的，不只是更强的模型，而是更好的环境、更稳的脚手架、更严格的评测，以及更完整的安全机制。未来决定 Agent 产品成败的，很可能不是谁最会说，而是谁最会稳稳地把事做成。
