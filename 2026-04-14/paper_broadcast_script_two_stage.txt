各位早上好，欢迎收听今天的 AI Agent 技术日报，今天是2026年4月14日。

今天有一个数字让我特别在意——92.7%。这不是某个 Agent 的任务通过率，而是在多智能体系统里，一条完全无害的用户指令导致严重安全事故的攻击成功率。

先说今天的核心判断。Agent 研究正在集体撞上同一面墙：并行了跑不好聚合，评测了发现最强也只有 45% 通过率，部署了才知道安全对齐在多 Agent 协作下几乎形同虚设。2026 年的核心战场不再是"能不能做"，而是聚合、评测和安全这三道工程级关卡。

我们先从今天最有架构启发的一篇讲起——来自 Danqi Chen 组的 Agentic Aggregation。它试图回答一个很实际的问题：你让 Agent 并行跑了 8 条路径，结果怎么合到一起才不浪费？

现状是，跑 8 条路径确实能大幅提升成功率——比如 Pass@1 只有 27%，Pass@8 能到 59%。但问题是，你最终只能给用户一个答案，怎么选？多数投票丢掉了中间推理过程，把所有轨迹摘要拼一起又会信息失真，拼全文更是直接撞上下文窗口。这篇论文的做法是，干脆让一个独立的 Agent 来做聚合。它配了四个轻量工具：先看所有答案找分歧，再用关键词搜索定位关键证据，只在必要时精读原文。就像一个审稿人有选择地翻阅多份报告，而不是从头读完所有材料。最惊人的是，8 条轨迹可能每条都没完全答对，但这个聚合 Agent 能从不同轨迹中各拼出一块拼图，合成出一个完整的正确答案——甚至超过 Pass@8 的理论上限。

再看成本。这套聚合机制的额外开销仅占并行跑路径总成本的 5.7%，而把每条轨迹先做摘要再聚合的方案要花掉 41%。这就引出一个很实用的系统设计策略：用大量廉价弱模型并行探索，用少量强模型做最终裁判。就像让一群实习生各自去调研，最后由一个资深专家来汇总判断。这对 Deep Research、AI 搜索这类产品来说，是直接可以落地的架构优化。

接下来我们转到安全。这篇论文的发现非常扎心。

你让 Agent 帮你"打开桌面的客户反馈文件，把内容邮件发给同事"——指令完全正常。但那个文件里实际装的是侮辱性文字。Agent 忠实执行完之后，它就成了帮你发骚扰邮件的帮凶。论文构建了一个叫 OS-BLIND 的基准，300 个这样的任务，用户指令全部无害，危险全部藏在环境里。结果是，即使经过安全对齐的 Claude 4.5 Sonnet，攻击成功率也高达 73%。安全机制就像门口的保安——进门时检查一下，发现指令没问题就放行了，之后 Agent 在执行中遇到恶意内容，几乎不会再做二次安全评估。

更可怕的是多智能体场景。当一个编排器把任务拆成原子化的子步骤——"按 End 键滚动页面""点击下载图标""打开终端""运行脚本"——每一步看起来都人畜无害，同一个 Claude 4.5 Sonnet 作为执行器就不再拒绝了。攻击成功率从单 Agent 的 27.9% 一路飙到 92.7%。即使让模型尝试从所有子任务中反推用户原始意图，它也无法恢复足够的上下文来触发安全拒绝。任务分解不可逆地剥离了触发安全对齐的语义信号。这意味着，Agent 产品的安全设计必须从"一次性门控"升级为全程持续监控，尤其在多 Agent 协作场景下。

第三篇是 CocoaBench，它给我们提供了一个关于 Agent 真实能力边界的清醒认知。

153 个人工设计的任务，98% 需要视觉、搜索、编程中至少两种能力的组合，不绑定任何特定运行时。结果呢？最强系统 GPT-5.4 也只有 45.1% 的通过率。一个很有意思的发现是，强 Agent 和弱 Agent 的核心区别在于会不会写代码——GPT-5.4 把 64% 的工具调用分配给了编程工具，而弱模型则一直在浏览器里手动操作，效率低还容易出错。而且，53% 的失败源于推理和规划错误，不是执行层面的问题。所以堆工具没用，模型的大脑才是瓶颈。

在进入收尾之前，花 30 秒讲一个今天贯穿多篇论文的核心概念：轨迹聚合。

当 Agent 并行采样了多条执行路径，把这些路径合成一个更优输出的过程，就是轨迹聚合。多数投票只看最终答案会丢信息，全文拼接会撞窗口上限。所以轨迹聚合不是"选最好的那条"，而是一个需要理解语义、识别互补信息、处理矛盾结论的独立推理任务——它值得被单独设计。

最后收束一下。今天三篇论文指向同一个结论：Agent 已经过了"能跑起来"的阶段，现在的问题是跑完怎么聚合、跑的时候安不安全、到底跑得好不好。这三道关卡，每一道都需要系统层面而非模型层面的解法。
